# DSA UI 重构方案

## 1. 结论摘要

目标目录：`apps/dsa-ui`

核心结论：

- 当前 `apps/dsa-web` 是一个独立的 React + Vite + TypeScript 单页前端。
- FastAPI 主要负责提供 API 和托管 `static/` 静态资源，前端不是服务端模板渲染。
- 因此 `apps/dsa-ui` 可以作为“新前端壳”独立开发，完全复用现有 API，不需要改后端接口契约。
- 现阶段最适合的方案不是继续在 `dsa-web` 上局部修补，而是以“信息架构 + 交互模型 + 视觉系统”三件事一起重做。

推荐方向：

- 保留 React + Vite + TypeScript。
- 使用 `Tailwind CSS v4 + Radix UI + shadcn/ui` 作为新的 UI 基础层。
- 用 `TanStack Query` 管理服务端状态，用 `Zustand` 管理局部跨页 UI 状态。
- 直接使用 `lightweight-charts` 作为图表基础能力，不再混用多套图表库。
- 采用“桌面端左侧工作台导航 + 移动端底部导航”的新壳层。
- 首页、持仓、设置三页全部重做信息组织，不再使用当前的“长页面堆叠式布局”。

当前落地状态（2026-03-27）：

- `apps/dsa-ui` 已覆盖分析台、问股、持仓、回测、设置、登录六大功能面。
- 问股已接入流式 SSE、Markdown 渲染、思考过程折叠与分析台追问上下文注入。
- 持仓/回测/设置与登录已完成任务型重构，保持后端 API 零改动。
- 构建切换支持并行预览与接管：`npm run build`（`static/`）与 `npm run build:preview`（`static-ui-preview/`）。

## 2. 现状审计

基于代码与本地运行态（`http://localhost:8000/`）确认，当前前端主要功能如下：

- 登录 / 首次密码设置
- 首页分析工作台
- 股票代码/名称输入与自动补全
- 提交分析任务
- 推送通知开关
- 实时任务流与历史分析列表
- 历史分析详情查看
- 完整 Markdown 报告查看 / 复制
- 从报告跳转到问股继续追问
- 问股多轮对话
- 策略选择（当前接口返回 11 个 skill）
- 问股历史会话切换 / 删除 / 导出 / 发送
- 持仓账户创建、组合快照、风险摘要
- 手工录入交易 / 资金流水 / 公司行为
- CSV 导入解析与提交
- 事件流水查看与删除
- 汇率刷新
- 回测运行、过滤、分页、绩效指标查看
- 系统设置（当前配置项 131 个）
- 认证设置、修改密码、桌面端 `.env` 导入导出
- 智能导入股票列表
- LLM 渠道编辑与连接测试

当前主要问题：

- 功能入口有，但页面结构不围绕“高频操作路径”组织。
- 首页、持仓、设置的信息层级过深，用户先看见的是组件堆叠，不是任务流。
- 视觉语言不统一：登录页偏炫技，业务页偏玻璃拟态，功能页又夹杂 terminal 风格。
- 移动端多数只是“压缩桌面布局”，不是“重新设计移动流程”。
- 长列表、长表单、长页面太多，导致认知负担高。
- 中英混用不稳定，例如回测页仍有较多英文控件文案。

## 3. 约束与架构判断

确认结论：

- 新 UI 应继续做成纯前端 SPA。
- API 继续走同源 `/api/v1/**`，不改后端接口。
- 开发阶段 `apps/dsa-ui` 可以跑独立 Vite dev server，通过代理转发到 `http://127.0.0.1:8000`。
- 切换上线时，只需要让新前端构建产物进入 `static/`，FastAPI 继续托管即可。

建议的切换方式：

- 联调阶段：使用 Vite dev server 直接访问新 UI。
- 预览构建：执行 `npm run build:preview` 输出到 `static-ui-preview/`，用于并行验证。
- 接管构建：执行默认 `npm run build`（或 `npm run build:static`）输出到根目录 `static/`，替换现有 `dsa-web` 构建产物。

这意味着：

- “不改后端”可以理解为“不改 API 与业务逻辑”。
- 真正需要动的只有前端工程和最终构建接管方式。

## 4. 推荐技术栈

### 4.1 推荐组合

核心框架：

- React 19
- Vite 7
- TypeScript
- React Router

UI 与样式：

- Tailwind CSS v4
- Radix UI
- shadcn/ui
- Lucide Icons
- Motion

状态与数据：

- TanStack Query：接口缓存、重试、并行请求、失焦重载控制
- Zustand：跨页面轻量 UI 状态，例如会话态、布局态、筛选态

表单与校验：

- React Hook Form
- Zod

表格与复杂交互：

- TanStack Table

图表：

- 直接使用 `lightweight-charts`
- 所有图表交互优先围绕金融时间序列、对比趋势、区间表现、柱状分布来设计
- 避免为了迁就旧页面继续保留饼图类表达，集中度与占比类信息改为排序条、堆叠条、热度列表或区间带

### 4.2 为什么不建议继续沿用当前组件层直接修补

- 现有 `dsa-web` 已经形成较多自定义样式和页面级状态耦合。
- 继续修补会把“旧信息架构”带进新项目。
- 新 UI 的重点不是换颜色，而是重建页面结构与交互流。

### 4.3 为什么推荐 `shadcn/ui`

- 轻，不会像重型设计系统那样把视觉强行锁死。
- 底层基于 Radix，交互基础能力成熟。
- 很适合做“看起来像现代工作台”的高效率界面。
- 可直接复制组件源码，便于针对本项目深度定制，而不是被库样式牵着走。

### 4.4 为什么直接采用 `lightweight-charts`

- 它更适合股票、回测、走势、区间表现等金融可视化场景。
- 运行时负担更轻，适合桌面和移动端都保持顺滑。
- 能减少图表层的技术分裂，避免后续维护多套图表组件体系。
- 可以把视觉风格统一到“专业交易工作台”而不是“通用数据大盘”。

设计上的同步调整：

- 持仓集中度不再用饼图，改为按权重排序的暴露条和 Top N 列表。
- 回测表现优先使用折线、面积、柱状和区间高亮。
- 首页报告中的行情片段、回测页的统计走势、持仓页的暴露变化全部复用同一图表语言。

## 5. 设计原则

- 功能不变，但操作路径更短。
- 先展示“现在该做什么”，再展示“所有能做什么”。
- 高优先级信息常驻，可选信息折叠。
- 移动端不是缩小桌面，而是重排任务流。
- 保持轻量、克制、明快，不走赛博炫光路线。
- 对风险写操作保持显式确认，例如删除流水、提交导入、覆盖配置。

## 6. 新的信息架构

一级导航保持不变：

- 分析台
- 问股
- 持仓
- 回测
- 设置

壳层建议：

- 桌面端：左侧窄导航栏 + 顶部上下文条 + 主工作区
- 平板端：左栏可折叠，主操作区优先
- 移动端：底部导航栏 + 页面内顶部操作条

移动端底部导航建议：

- 分析
- 问股
- 持仓
- 回测
- 设置

不建议在移动端继续使用默认汉堡菜单作为唯一主导航入口。

## 7. 页面级功能设计

### 7.1 分析台

目标：让“输入股票 -> 发起分析 -> 看报告 -> 继续追问”成为一条最短路径。

重构后的页面结构：

- 顶部固定分析条：股票搜索、快捷建议、通知开关、开始分析
- 左侧信息栏：进行中任务 + 历史分析
- 主区：当前选中报告
- 报告内部用 Tab 组织，而不是整页长滚动

报告内容建议拆成：

- `总览`
- `策略点位`
- `资讯`
- `透明度`
- `Markdown`

交互优化：

- 历史列表默认单击查看，批量删除进入“管理模式”后再显示多选。
- 选中报告后，右上固定显示 `追问 AI`、`查看 Markdown`、`复制摘要`。
- 当任务完成时，历史列表自动插入新报告并轻提示，不强制打断当前阅读。

### 7.2 问股

目标：把它做成“可追问的策略工作区”，不是普通聊天框。

新结构：

- 左侧：会话列表
- 主区：消息流
- 底部：输入区
- 顶部：当前会话标题、导出、发送、上下文状态

关键改动：

- 技能选择改为 `策略下拉 + 热门策略 chips`，不再整排 radio 堆满输入区。
- 当从报告页进入时，输入框上方显示“已附带上一份分析上下文”的状态条。
- 工具调用过程改为更清晰的折叠时间线。
- 会话操作收敛到 `更多` 菜单，减少头部拥挤。

### 7.3 持仓

这是最需要重新拆页的页面。

当前问题：

- 读操作和写操作混在一页。
- 账户切换、总览、录入、导入、流水全部堆叠，过长。
- 在移动端尤其不友好。

建议拆成二级标签：

- `总览`
- `持仓`
- `录入`
- `导入`
- `流水`

每个标签职责：

- `总览`：总权益、总市值、总现金、汇率状态、集中度、回撤、止损预警
- `持仓`：持仓表、账户过滤、成本口径切换
- `录入`：交易 / 资金 / 公司行为三种录入，用分段控件切换
- `导入`：CSV 解析、预演结果、提交结果，做成三步式流程
- `流水`：交易 / 资金 / 公司行为记录，支持筛选、分页、删除

关键交互：

- 当处于“全部账户”时，写操作入口保留但置灰，并明确提示“请选择具体账户后再写入”。
- 录入表单不要三列并排塞满页面，桌面端用卡片切换，移动端用步骤流。
- 导入结果必须先预览，再允许提交。
- 暴露与集中度不再依赖饼图，直接展示 `Top exposures` 排序条、账户分布带和风险提示列表。

### 7.4 回测

目标：让用户一眼看到“模型最近准不准”。

建议结构：

- 顶部控制条：股票过滤、评估窗口、强制回测、运行按钮
- KPI 区：方向准确率、胜率、平均模拟收益、平均股票收益
- 对比区：全局表现 vs 当前股票表现
- 结果区：表格 + 分页

改动重点：

- 文案统一中文。
- 把“运行回测”和“查看结果”分成两个视觉层级。
- 表格支持更清晰的 outcome/status 标签色。
- 使用 `lightweight-charts` 展示全局表现趋势、单股表现对比、命中分布和窗口切换后的曲线重绘。

### 7.5 设置

当前问题：

- 131 个配置项放在一个长表单体系里，查找成本高。
- 左栏只是分类，不支持搜索、最近修改、脏状态聚合。

建议结构：

- 顶部：搜索配置项、脏数据提示、保存/重置
- 左侧：分类导航
- 主区：分类概览卡片 + 表单
- 底部：脏数据时出现粘性保存条

重点功能模块独立成专属面板：

- `LLM 渠道与模型`
- `认证与密码`
- `智能导入`
- `桌面端 env 备份`

交互优化：

- 增加配置项搜索。
- 支持“只看已修改”。
- 分类页顶部展示本分类说明和风险级别。
- 对敏感字段统一采用 reveal 机制和保存确认。

### 7.6 登录 / 认证

建议保留独立登录页，但视觉上回归简洁。

目标：

- 比当前更稳、更干净
- 保留品牌感，但弱化重背景和大面积动态效果
- 首次设置密码与普通登录共用同一结构

## 8. 响应式策略

### 8.1 断点策略

- `>= 1280px`：桌面工作台
- `768px - 1279px`：平板 / 小屏桌面
- `< 768px`：手机

### 8.2 移动端设计规则

- 一级导航用底部栏，不用左抽屉作为唯一方案。
- 搜索、发送、保存等高频动作固定在拇指容易触达的位置。
- 长表格优先改成卡片列表，必要时再允许横向滚动。
- 二级筛选和详情适合用 Drawer / Bottom Sheet。
- 分析台在手机上改成 `报告 / 历史` 双视图切换，而不是同时塞进一屏。

---

## 9. 视觉设计方向

视觉关键词：

- 轻
- 亮
- 干净
- 高效率
- 面向数据决策

建议风格：

- Light first，Dark mode 可选，不再以深色赛博感作为默认人格
- 主色建议使用 `teal / emerald` 体系
- 风险与收益状态使用更明确的市场语义色
- 大面积留白 + 低对比卡面 + 高对比数据数字

避免继续沿用的风格：

- 大量紫青霓虹混搭
- 不必要的玻璃拟态
- 登录页和业务页像两个产品
- 所有页面都使用相同强烈渐变边框

字体建议：

- UI 主字体：`Geist` 或 `Plus Jakarta Sans`
- 中文 fallback：`Noto Sans SC` / `PingFang SC`
- 数据数字：`JetBrains Mono`

## 11. `apps/dsa-ui` 建议目录结构

```text
apps/dsa-ui/
  src/
    app/
      router/
      providers/
      layouts/
    features/
      auth/
      dashboard/
      chat/
      portfolio/
      backtest/
      settings/
    shared/
      api/
      types/
      ui/
      hooks/
      utils/
      styles/
    main.tsx
```

目录设计原则：

- 页面和业务模块分离，不再按“组件堆”组织。
- `shared/api` 初期可直接复用或迁移 `dsa-web/src/api` 与 `src/types`。
- 旧前端与新前端并行存在，避免重构过程影响现网可用性。

## 13. 实施节奏建议

### Phase 1：搭壳

- 初始化 `apps/dsa-ui`
- 建立主题、路由、导航壳层
- 接好 API client、Query provider、表单基础设施

### Phase 2：先做高频页

- 分析台
- 问股

原因：

- 这是当前用户最常用的闭环
- 也是最能体现“新 UI 价值”的两页

### Phase 3：重做长页

- 持仓
- 回测

### Phase 4：配置后台

- 设置
- 登录 / 认证

### Phase 5：切换与收尾

- 构建输出切换到 `static/`
- 更新脚本和文档
- 保留 `dsa-web` 一段时间作为回滚方案

---

## 14. 风险点

- 设置页配置项多，必须优先做好搜索、分组和 dirty state 提示。
- 持仓页涉及写操作，不能只做视觉重构，必须重建防误操作路径。
- 分析台与问股都依赖异步任务和流式状态，状态管理不能再散落在页面内部。
- 若一开始就强行做全量视觉细节，容易拖慢交付，建议先完成结构重构，再补品牌细节。

---

## 15. 最终建议

建议直接立项 `apps/dsa-ui`，不要继续在 `apps/dsa-web` 上做“大修式迭代”。

最优先落地顺序：

- 先做新壳层
- 再做分析台
- 再做问股
- 然后做持仓和设置

如果进入实现阶段，第一版目标不是“更炫”，而是：

- 更快找到功能
- 更短完成任务
- 更少误操作
- 更适合手机使用
- 更容易继续演进
