# DSA UI 开发计划

## 1. 目标

本计划用于指导 `apps/dsa-ui` 从方案阶段进入可执行开发阶段，并明确：

- 开发阶段划分
- 任务拆解
- 每阶段交付物
- 每阶段测试门禁
- 浏览器级验证要求
- 可中断、可恢复的执行规则

本计划与下列文档配套使用：

- [DSA UI 重构方案](./redesign-proposal)
- [DSA UI 任务跟踪](./task-tracker)

---

## 2. 开发原则

- 不改后端 API，不新增平行后端实现。
- 优先迁移和复用已有类型、API client、认证与错误处理能力。
- 先完成结构与交互，再补视觉细节。
- 每个阶段必须有测试闭环，测试未过不得标记完成。
- 每次开发中断前必须更新任务跟踪文档。

---

## 3. 技术基线

前端框架：

- React 19
- Vite 8
- TypeScript
- React Router 7

UI 基础：

- Tailwind CSS v4
- 自定义设计令牌与原子组件
- Motion
- Lucide
- React Markdown + remark-gfm

状态与数据：

- TanStack Query
- Zustand
- React Hook Form
- Zod

图表：

- `lightweight-charts`

测试：

- `Vitest`：单元测试
- `@testing-library/react`：组件与集成测试
- `MSW`：接口模拟
- `Playwright`：浏览器级无头测试
- `axe-core/playwright`：基础可访问性检查
- `Lighthouse CI` 或等效性能预算检查：样式、性能、稳定性门禁

工程运行约束：

- 本地命令统一使用 `eval "$(fnm env)" && <command>`，确保 Node 22 环境。
- 浏览器级测试默认通过 `playwright.config.ts` 固化本地代理绕过，避免 `HTTP_PROXY`/`HTTPS_PROXY` 干扰 `127.0.0.1` 探测。
- 浏览器测试命令必须串行执行，避免多个 Playwright 进程并发抢占本地 web server 导致 `ERR_CONNECTION_REFUSED` 假失败。

---

## 4. 完成定义

任意任务只有同时满足以下条件才算完成：

- 代码实现完成
- 相关文档已同步
- 单元测试已补齐并通过
- 集成测试已补齐并通过
- 对应浏览器级测试已补齐并通过
- 无头浏览器下桌面与移动视口均验证通过
- 无新增控制台错误
- 无阻断性样式回归
- 性能和交互预算未超标

任意阶段只有在该阶段所有任务满足以上条件后，才可进入下一阶段。

## 5. 阶段计划

## Phase 0：工程基线

目标：

- 搭建 `apps/dsa-ui` 基础工程
- 接入统一测试体系
- 建立可持续的任务跟踪机制

任务：

- `UI-001` 初始化 Vite + React + TypeScript 项目
- `UI-002` 建立目录结构、路径别名、基础 lint/test 命令
- `UI-003` 建立主题 token、字体、颜色与布局基础组件
- `UI-004` 接入 React Router、应用壳层、桌面/移动端导航骨架
- `UI-005` 迁移或封装现有 API client 与通用 types
- `UI-006` 建立 TanStack Query、Zustand、表单基础设施
- `UI-007` 接入 `lightweight-charts` 基础封装组件
- `UI-008` 建立测试基线：Vitest + RTL + MSW + Playwright + axe + 性能检查
- `UI-009` 建立测试夹具与稳定数据种子

阶段交付物：

- 可运行的 `apps/dsa-ui`
- 基础路由壳层
- 测试命令与测试目录
- 首个浏览器 smoke 测试

阶段门禁：

- 单元测试：基础工具、hooks、stores、chart adapter
- 集成测试：路由壳层、认证重定向、错误状态
- 浏览器测试：桌面/移动端打开首页壳层、导航切换、主题切换
- 样式检查：关键壳层截图基线
- 性能检查：首屏壳层构建产物可在预算内加载

## Phase 1：分析台

目标：

- 完成首页分析工作台重构

任务：

- `UI-101` 顶部分析条：股票搜索、通知开关、提交分析
- `UI-102` 左侧任务栏与历史分析列表：活跃任务、状态同步、历史切换
- `UI-103` 报告头部与操作条：核心摘要、复制摘要、查看 Markdown、追问 AI
- `UI-104` 报告 Tabs：总览、策略点位、资讯、透明度、Markdown
- `UI-105` 历史管理模式：多选、全选、批量删除
- `UI-106` 移动端双视图切换：报告 / 历史
- `UI-107` Markdown 查看复制与追问跳转联动

阶段门禁：

- 单元测试：数据格式化、历史选择状态、报告 tab 状态、输入校验
- 集成测试：分析提交流程、历史切换、删除流程、Markdown 打开与复制
- 浏览器测试：桌面与移动端完整流程
- 样式检查：报告页关键断点截图
- 性能检查：报告切换、列表滚动、drawer 打开无明显卡顿

## Phase 2：问股

目标：

- 完成问股工作区重构

任务：

- `UI-201` 会话列表：切换、新建、删除
- `UI-202` 会话头部：导出、发送、上下文状态
- `UI-203` 输入区：策略下拉、快捷策略 chips、快捷发送
- `UI-204` 消息流渲染：Markdown、思考过程折叠、复制消息
- `UI-205` SSE/流式状态的视觉反馈
- `UI-206` 从分析台进入问股时的上下文注入
- `UI-207` 移动端聊天输入与历史面板交互

阶段门禁：

- 单元测试：会话状态管理、复制状态、上下文构建
- 集成测试：会话切换、流式消息、导出与发送、上下文追问
- 浏览器测试：桌面/移动端问股全流程
- 样式检查：消息气泡、长文渲染、折叠状态截图
- 性能检查：长会话滚动、流式渲染、移动端输入响应

## Phase 3：持仓

目标：

- 将持仓页从“大杂烩长页”拆成任务型工作区

任务：

- `UI-301` 账户选择、成本法、刷新等顶层控制
- `UI-302` 总览页：总权益、总市值、总现金、汇率状态
- `UI-303` 风险模块：回撤、止损预警、口径说明
- `UI-304` 集中度视图：改造为排序条、Top exposures、账户分布
- `UI-305` 持仓表：筛选、空状态、移动端卡片化
- `UI-306` 手工录入：交易 / 资金 / 公司行为分段表单
- `UI-307` CSV 导入：选择文件、解析、预演、提交三步式流程
- `UI-308` 流水页：交易 / 资金 / 公司行为列表、筛选、分页、删除
- `UI-309` 写操作保护：全部账户视图禁写、显式确认

阶段门禁：

- 单元测试：表单校验、写保护逻辑、数据映射
- 集成测试：账户切换、录入提交、导入预演、删除流水
- 浏览器测试：桌面/移动端完整持仓流程
- 样式检查：表格、卡片、导入流程、错误提示截图
- 性能检查：大表滚动、标签切换、图表更新

## Phase 4：回测

目标：

- 将回测页重做成“可读、可筛、可比较”的性能工作台

任务：

- `UI-402` KPI 卡片
- `UI-403` 全局表现趋势图
- `UI-404` 单股 vs 全局对比图
- `UI-405` 结果表格：分页、空状态、状态标签
- `UI-406` 移动端回测布局

阶段门禁：

- 单元测试：指标格式化、筛选参数构建、分页行为
- 集成测试：运行回测、获取结果、性能卡片与表格联动
- 浏览器测试：桌面/移动端回测流程
- 样式检查：图表与表格截图
- 性能检查：图表绘制、筛选切换、长表分页

## Phase 5：设置与登录

目标：

- 完成设置后台与认证页重构

任务：

- `UI-501` 设置壳层：搜索、分类导航、粘性保存条
- `UI-502` 分类面板与字段分组

- `UI-505` 智能导入
- `UI-506` 桌面端 env 导入导出
- `UI-507` 登录页 / 首次密码设置页
- `UI-508` 设置页错误、冲突、重试、脏数据恢复

阶段门禁：

- 单元测试：字段映射、冲突处理、脏数据状态
- 集成测试：保存、重置、导入导出、登录、首次设置密码
- 浏览器测试：桌面/移动端设置与登录流程

## 6. 测试矩阵

### 6.1 单元测试

覆盖范围：

- utils
- hooks
- stores
- 表单校验
- API adapter
- 图表数据转换器
- 布局状态与权限保护逻辑

要求：

- 每个新增核心模块必须有对应单元测试
- bug 修复必须补回归测试

### 6.2 集成测试

覆盖范围：

- 页面内多组件协作
- Query + store + form 联动
- API 错误状态、加载态、空态
- 导入/删除/保存等危险操作流程

要求：

- 使用 `MSW` 稳定模拟接口
- 用固定夹具保证结果可复现

### 6.3 浏览器级测试

工具：

- `Playwright` 无头模式

覆盖要求：

- 每个功能点必须至少有一个浏览器级 happy path
- 每个高风险功能点必须有失败场景测试
- 必须覆盖桌面视口与手机视口

必须覆盖的功能类别：

- 导航与路由
- 首页分析
- 历史分析切换与删除
- Markdown 打开、复制
- 问股发送、切换会话、删除会话、导出
- 持仓新建账户、录入、导入、删除流水
- 回测运行与筛选
- 设置保存、重置、搜索、认证、导入导出
- 登录与首次设置密码

### 6.4 样式与交互检查

浏览器级测试除功能外，必须同时检查：

- 关键页面截图基线
- 不同断点下无布局错乱、无文字溢出、无水平滚动异常
- drawer / dialog / sticky bar / bottom nav 交互正确
- 无阻断性控制台错误
- 主操作按钮始终可见且可触达

### 6.5 性能检查

性能检查基于生产构建后的预览环境执行。

最低要求：

- 核心页面首屏加载在预算内
- 页面切换与高频交互无明显卡顿
- 大列表滚动平滑
- 图表渲染和重绘不会造成阻断性交互抖动

建议门禁：

- 使用 `Lighthouse CI` 或等效脚本记录页面性能
- 为首页、问股、持仓、回测建立基准值
- 记录 bundle 体积变化

### 6.6 阶段测试门禁

每个阶段结束时必须至少执行：

```bash
eval "$(fnm env)"
npm run lint
npm run test:unit
npm run test:integration
npm run test:e2e
npm run test:visual
npm run test:perf
npm run test:e2e:backend
npm run build
```

说明：

- `test:e2e`、`test:visual`、`test:perf`、`test:e2e:backend` 必须串行执行。
- 如果命令名后续调整，必须同步更新本计划文档和任务跟踪文档。

## 7. 中断恢复规则

任务中断前必须做完以下动作：

- 更新 [DSA UI 任务跟踪](./task-tracker)
- 写明当前阶段、当前任务 ID、实际完成度
- 写明已修改文件
- 写明最后一次通过和失败的测试命令
- 写明下一步要执行的第一条命令
- 写明阻塞点和待决事项

恢复任务时必须先做：

- 先读任务跟踪文档的 `当前状态` 与 `最近交接记录`
- 再读本计划文档对应阶段门禁
- 再决定继续编码还是先补测试
